# マルチカメラ三角測量の知見

## sanitize_pose は不要

- キーポイントベースのクロップ方式では、画像端の検出も正当な検出
- 境界リジェクション（x<=0.02, x>=0.98）はクロップ前提なら無意味
- 品質管理はconfidenceスコアで行うべき
- 削除済み（PR #64）

## One Euro Filter の実効性

inference_server.toml の設定: `position_min_cutoff=4.5`, `beta=0.5`

- 15FPSでのNyquist周波数は7.5Hz
- min_cutoff=4.5はNyquistに近く、ほぼフィルタリングしていない
- 実測でfiltered/raw の標準偏差比が0.90〜1.00 → フィルタの効果が薄い
- min_cutoffを下げるとジッタは減るが動き出し遅延が発生するトレードオフ

## 2カメラペア固定の三角測量（旧方式、issue #67で廃止）

- `find_hip_reference_pair`で腰のreproj errorが最小の2カメラペアを選択
- 選択されたペアを全キーポイントに適用（ヒステリシス付き）
- cam3がペアに選ばれにくい → cam3が有効活用されていない
- 安定性は高いが、カメラ台数を増やしても恩恵を受けにくい構造
- **reproj errorがピクセル単位だったため、fx が小さいカメラ（cam3）のエラーが過小評価され、96.5%でcam0-cam3ペアが選ばれてしまう問題があった**

## 全カメラ活用の三角測量（試行→revert）

### 実装内容
1. 全カメラで三角測量
2. reproj errorが閾値超のカメラを除去して再測量
3. フォールバックとしてベストペアを使用

### 失敗した理由
- cam3のキャリブレーション品質が低く（fx=2545、期待値~1500）、ノイズ源になった
- 全カメラ結果とペアフォールバック結果の間に不連続が発生
- 動作時に下半身が崩壊（R_footが132cm跳ぶ、hipがz=0.78mに飛ぶ等）
- 画像端・前方への移動時に特に顕著

### 教訓
- 堅牢なマルチカメラ三角測量の前提条件として、各カメラのキャリブレーション品質が十分であること
- キャリブレーション品質が悪いカメラを含めると、2カメラより悪化する
- 段階的フォールバックは結果の不連続を生みやすい

## cam3のキャリブレーション問題

### 異常パラメータ（修正前）
- fx=2545.8（1920px幅で期待値は1000〜1800）
- dist_coeffs: k1=-0.67, k2=5.49, k3=-15.16（極端）

### 設置条件の課題
- cam0の真上（高さ~2m）から見下ろす急俯角
- ボードの観測条件が悪くなりやすい（斜めに映る、小さく映る）
- z方向の精度向上には不可欠な配置だが、キャリブレーション難度が高い

### 改善方針（issue #65）
- ボードをcam3に近づけて大きく映す
- 俯角に合わせてボードを傾ける
- reproj error < 0.3 を目標
- 品質改善後にマルチカメラ三角測量を再実装

## キーポイントごとの全カメラDLT + 外れカメラ除外（issue #67）

### 変更内容
1. `reprojection_error`をピクセル→角度ベース（`atan(px_err/fx)` ラジアン）に変更
2. `MAX_REPROJ_ANGLE = 0.084 rad`（旧 `MAX_REPROJ_ERROR = 120px` at fx=1420相当）
3. `find_hip_reference_pair`とペア固定ロジックを削除
4. キーポイントごとに全カメラでDLT → 角度reproj error閾値超のカメラを除外 → 2台以上残れば再三角測量

### 前回の試行（revert済み）との違い
- reproj errorが角度ベースになり、fx依存のバイアスが解消
- ペアフォールバック（不連続の原因）を廃止し、inlier/outlier判定のみ
- cam3のキャリブレーション品質改善（PR#66）が前提条件

### 制約・既知の限界
- `atan(px_err/fx)` は fx のみで正規化するため、fx≠fy のカメラでは方向依存の近似
- 外れカメラが p3d を汚染するケース：外れカメラの影響が強すぎると正常カメラもthreshold超で全drop → キーポイント棄却（壊れた座標を出すよりはマシ）
